Task Manager — API REST

Gerenciador de tarefas com Spring Boot, organizado em camadas (domain, application, infrastructure) no estilo DDD. Cada operação passa por um use case dedicado; a persistência é em memória com HashMap, sem banco de dados — foco em praticar injeção de dependência, REST e separação de responsabilidades antes de evoluir para JPA e Security na trilha.

O que é este projeto

Continuação natural dos exercícios de POO no terminal: a mesma ideia de domínio no centro e repositório trocável, agora exposta por HTTP com JSON. O recurso central é a Task — título, descrição opcional e status (PENDING, IN_PROGRESS, COMPLETED).

Deliberadamente sem banco de dados: o InMemoryTaskRepository guarda tarefas em um Map<TaskId, Task> para isolar a regra de negócio da infraestrutura e praticar o contrato TaskRepository antes de plugar Postgres ou JPA.

Início rápido

Projeto Gradle com Spring Boot. Sobe na porta padrão 8080 — sem Docker nem banco.

  • 1. Java 25 (toolchain definida no build.gradle)
  • 2. Na pasta do repositório, subir a aplicação
  • 3. Testar os endpoints em http://localhost:8080/tasks
# Subir a API
./gradlew bootRun

# Windows
gradlew.bat bootRun

# Rodar testes
./gradlew test

Exemplo — criar tarefa: 

curl -X POST http://localhost:8080/tasks \
  -H "Content-Type: application/json" \
  -d '{"title":"Estudar Spring Boot","description":"Módulo REST"}'

Endpoints REST

Um único controller (TaskController) concentra todas as rotas em /tasks. Ele recebe DTOs HTTP, converte para input da camada application e devolve TaskResponseDTO em JSON.

  • POST /tasks — cria tarefa (status inicial PENDING)
  • GET /tasks — lista todas as tarefas
  • GET /tasks/{id} — busca por UUID
  • PATCH /tasks/{id} — atualização parcial (título, descrição e/ou status)
  • DELETE /tasks/{id} — remove tarefa (204 No Content)
// Resposta JSON (exemplo)
{
  "id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "title": "Estudar Spring Boot",
  "description": "Módulo REST",
  "status": "PENDING"
}

Atualização parcial via PATCH — campos omitidos não são alterados: 

curl -X PATCH http://localhost:8080/tasks/{id} \
  -H "Content-Type: application/json" \
  -d '{"status":"IN_PROGRESS"}'

Arquitetura em camadas (DDD)

Três pacotes principais, com o domínio independente de Spring e HTTP: 

dio.taskmanager/
├── domain/
│   ├── Task.java              # entidade + regras de update
│   ├── TaskId.java            # value object (UUID)
│   ├── TaskStatus.java        # PENDING | IN_PROGRESS | COMPLETED
│   └── TaskRepository.java    # contrato (interface)
├── application/
│   ├── CreateTaskUseCase.java
│   ├── GetTaskUseCase.java
│   ├── GetTaskByIdUseCase.java
│   ├── UpdateTaskUseCase.java
│   ├── DeleteTaskUseCase.java
│   └── input/ · output/       # DTOs da camada de aplicação
└── infrastructure/
    ├── repository/
    │   └── InMemoryTaskRepository.java   # HashMap em memória
    └── http/
        ├── TaskController.java           # todas as rotas REST
        ├── GlobalExceptionHandler.java
        └── request/ · response/          # DTOs HTTP
  • Domain: Task valida título e encapsula mudança de estado via update()
  • Application: um use case por operação — orquestra domínio + repositório
  • Infrastructure: implementação concreta do repositório e adaptadores HTTP (request/response DTOs)
  • Controller: só traduz HTTP ↔ use cases — sem regra de negócio

Decisões técnicas

Por que use case em vez de um service monolítico?

Cada operação (criar, listar, buscar, atualizar, deletar) tem uma classe própria anotada com @Service. Isso deixa a intenção explícita, facilita testes isolados com Mockito e espelha o padrão que vi nos exercícios de POO — só que com Spring cuidando da injeção.

Dois tipos de DTO — HTTP e application

CreateTaskRequestDTO e UpdateTaskRequest ficam em infrastructure/http; CreateTaskInputDTO e UpdateTaskInputDTO ficam em application/input. O controller converte request → input; o use case nunca depende de anotações web.

Repositório em memória com HashMap

InMemoryTaskRepository implementa TaskRepository com Map<TaskId, Task>. Mesma abordagem do exercício 3 do bootcamp POO — trocar a implementação depois (JPA, JDBC) sem alterar os use cases.

TaskId como value object

TaskId é um record que encapsula UUID e gera id automaticamente no construtor vazio. O domínio não expõe UUID cru — a infraestrutura converte na borda do controller.

Testes

Cobertura focada em regras e persistência em memória, sem subir o contexto Spring inteiro nos testes unitários principais:

  • CreateTaskUseCaseTest — valida campos do output (título, descrição, status PENDING, id gerado)
  • InMemoryTaskRepositoryTest — save, findAll, findById e delete no HashMap
  • TaskmanagerApplicationTests — smoke test de contexto Spring Boot
./gradlew test

Stack

  • Java 25 + Gradle
  • Spring Boot 4.1 — starter web, embedded Tomcat, DI
  • Lombok — getters/setters na entidade Task
  • JUnit 5 + Mockito + AssertJ — testes unitários
  • Sem banco — persistência em memória apenas

Próximos passos na trilha do bootcamp: Bean Validation, OpenAPI/Swagger, Spring Data JPA e Security — este repositório cobre o núcleo REST + camadas.

Relação com os outros projetos do bootcamp

Os exercícios de POO no terminal introduziram domínio, interface de repositório e service. O JDBC Sample mostrou persistência real com SQL. Este Task Manager junta HTTP (Spring) com a mesma organização em camadas — passo natural antes de trocar o HashMap por Postgres ou JPA.
