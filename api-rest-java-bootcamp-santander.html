API REST — Gerenciador de tarefas

Próxima etapa do bootcamp depois dos exercícios de POO: construir uma API REST com Spring Boot para gerenciar tarefas. O repositório no GitHub já está criado; a implementação ainda vai começar — esta página documenta o objetivo, a trilha do curso e como pretendo organizar o projeto.

O que é este projeto

Enquanto os exercícios de POO rodam no terminal, esta trilha leva o mesmo tipo de regra de negócio para a web: endpoints HTTP, status codes, corpo JSON e separação entre controller, serviço e domínio — agora com Spring Boot cuidando do servidor embarcado, injeção de dependência e convenções REST.

O caso de uso central é um gerenciador de tarefas: criar, listar, consultar, validar e evoluir tarefas via API, em vez de menus no Scanner.

Status: repositório criado, código ainda não iniciado. A documentação aqui será atualizada conforme cada módulo do bootcamp for concluído.

Trilha do curso — módulos

A formação de API REST no bootcamp Santander · DIO segue esta sequência. Conforme for avançando, marco aqui o que já está no repositório.

  • Introdução ao API REST — conceitos HTTP, recursos e verbos
  • Gerenciamento de tarefas — escopo do projeto e requisitos
  • Modelando o domínio de tarefas — entidades e regras de negócio
  • Orquestrando o domínio — camada de serviço / application
  • Listagem de tarefas — endpoint GET com coleção
  • Infraestrutura e interface — persistência e controllers REST
  • Consulta de tarefas — busca por id e filtros
  • Validando dados — Bean Validation e respostas de erro
  • Documentando a API — OpenAPI / Swagger
  • Evoluindo a API — melhorias, versionamento e boas práticas

Como pretendo organizar o código

A ideia é manter a mesma linha dos exercícios de POO — domínio no centro, infraestrutura trocável — só que exposto por controllers Spring em vez de UI no terminal. 

api-rest-java-bootcamp-santander/
├── src/main/java/.../
│   ├── domain/           # Task, regras de negócio
│   ├── application/      # casos de uso / orquestração
│   ├── infrastructure/   # repositório (memória ou banco)
│   └── web/              # controllers REST, DTOs, validação
├── src/main/resources/
│   └── application.properties
└── build.gradle.kts      # Spring Boot + dependências
  • Controllers: recebem HTTP, delegam ao service, devolvem JSON e status adequados
  • Service: orquestra o domínio — equivalente ao PetShopService do exercício 3
  • Domain: entidade Task com regras (título, status, prazo, etc.)
  • Infrastructure: persistência — primeiro em memória ou JPA, conforme o módulo

Stack prevista

  • Java 17+ e Gradle
  • Spring Boot — starter web, embedded Tomcat, DI
  • Spring Web — @RestController, @GetMapping, @PostMapping, etc.
  • Bean Validation — @NotBlank, @Valid nos DTOs de entrada
  • Documentação — SpringDoc / OpenAPI quando chegar no módulo correspondente

Depois desta base REST, o bootcamp continua com Spring Data, Security e OpenFeign — este repositório foca primeiro no núcleo da API de tarefas.

Relação com os outros projetos do bootcamp
