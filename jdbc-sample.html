JDBC Sample — CRUD com PostgreSQL

Projeto de estudo para entender JDBC na prática: banco PostgreSQL rodando no Docker, schema versionado com Flyway, CRUD completo de funcionários (employees) e auditoria automática via triggers no banco — tudo sem Spring ou ORM. Feito para consulta rápida e para mostrar a recrutadores como configurei cada camada do zero.

Início rápido Como fiz o CRUD Triggers de auditoria
Java JDBC PostgreSQL Flyway Triggers Docker Gradle

Início rápido

O fluxo é direto: subir o Postgres no Docker, rodar a aplicação Java e deixar o Flyway criar a tabela automaticamente antes de executar o CRUD.

  • 1. Docker Desktop instalado e rodando
  • 2. Java 17+ (Gradle 9 exige JVM 17 ou superior)
  • 3. Na pasta do projeto, subir o banco e rodar a app (comandos abaixo)
# 1. Subir o banco
docker compose up -d

# 2. Executar a aplicação (migrations + CRUD)
./gradlew run

# Windows
gradlew.bat run

# Conferir se o container está ativo
docker ps

Porta do banco: 5433 no host (mapeada para 5432 dentro do container). Escolhi 5433 para não conflitar com um Postgres local na porta padrão.

1. Docker — criar e configurar o banco

Usei Docker Compose para subir um PostgreSQL 16 com banco, usuário e senha já definidos. O volume postgres_data persiste os dados entre reinícios do container. 

# docker-compose.yml
services:
  postgres:
    image: postgres:16
    ports:
      - "5433:5432"
    environment:
      POSTGRES_DB: jdbc-sample
      POSTGRES_USER: postgres
      POSTGRES_PASSWORD: 050495
    volumes:
      - postgres_data:/var/lib/postgresql/data

volumes:
  postgres_data:
  • POSTGRES_DB: nome do banco que a aplicação usa na URL JDBC
  • POSTGRES_USER / POSTGRES_PASSWORD: mesmas credenciais em ConnectionUnit e no Flyway
  • volume: dados não se perdem ao parar o container (docker compose down sem -v)

Se mudar a senha depois da primeira subida, é preciso alterar no Postgres também (o volume já guardou a senha antiga): 

docker exec -it jdbc-sample-postgres-1 psql -U postgres \
  -c "ALTER USER postgres WITH PASSWORD '050495';"

2. Gradle — dependências JDBC e Flyway

No build.gradle.kts adicionei o driver PostgreSQL, o Flyway core e o módulo específico para Postgres. O plugin application aponta o Main como entry point. 

plugins {
    id("java")
    application
}

application {
    mainClass.set("org.example.Main")
}

dependencies {
    implementation("org.flywaydb:flyway-core:10.11.1")
    implementation("org.flywaydb:flyway-database-postgresql:10.11.1")
    runtimeOnly("org.postgresql:postgresql:42.7.7")
}

3. Flyway — migration na Main

Antes de qualquer operação JDBC, a Main configura o Flyway com a mesma URL do banco e chama migrate(). O Flyway lê os arquivos em src/main/resources/db/migration/ e aplica só o que ainda não rodou. 

var flyway = Flyway.configure()
        .dataSource(
            "jdbc:postgresql://localhost:5433/jdbc-sample",
            "postgres",
            "050495")
        .load();
flyway.migrate();

O Flyway aplica quatro migrations em ordem:

  • V1 — tabela employees
  • V2 — tabela employees_audit para histórico
  • V3 — trigger AFTER INSERT na employees
  • V4 — trigger AFTER UPDATE na employees

A V1 cria a tabela principal:

-- V1__create_employee_table.sql
CREATE TABLE employees (
    id BIGSERIAL PRIMARY KEY,
    name VARCHAR(150) NOT NULL,
    salary DECIMAL(10, 2) NOT NULL,
    birthday TIMESTAMPTZ NOT NULL
);

Convenção Flyway: prefixo V + número + __ + descrição. Cada alteração de schema vira um novo arquivo numerado — o Flyway só executa o que ainda não rodou.

4. Conexão JDBC — ConnectionUnit

Centralizei a URL, usuário e senha em uma classe utilitária com construtor privado. Cada operação do DAO abre a conexão com try-with-resources e fecha automaticamente. 

public class ConnectionUnit {
    private ConnectionUnit() {}

    public static Connection getConnection() throws SQLException {
        return DriverManager.getConnection(
            "jdbc:postgresql://localhost:5433/jdbc-sample",
            "postgres",
            "050495");
    }
}

Em produção eu extrairia isso para variáveis de ambiente ou arquivo de configuração. Aqui mantive fixo de propósito para focar no JDBC puro.

5. CRUD completo — Entity + DAO

Separei a entidade (EmployeeEntity) da persistência (EmployeeDAO). Sem framework: SQL escrito à mão, PreparedStatement com placeholders (?) e mapeamento manual do ResultSet. O DAO expõe cinco operações: insert, update, delete, findAll e findById.

Estrutura do projeto

jdbc-sample/
├── docker-compose.yml
├── build.gradle.kts
├── src/main/java/org/example/
│   ├── Main.java

│       └── entity/EmployeeEntity.java
└── src/main/resources/db/migration/

    └── V4__create_trigger_update_employee.sql

Entity

Classe POJO com getters/setters para id, name, salary e birthday (OffsetDateTime para bater com TIMESTAMPTZ).

Insert — retorna o id gerado

Uso RETURNING id no PostgreSQL para recuperar o BIGSERIAL gerado e preencher o objeto em memória. Cada insert dispara o trigger de auditoria (seção 6). 

var sql = "INSERT INTO employees (name, salary, birthday) VALUES (?, ?, ?) RETURNING id";

try (var connection = ConnectionUnit.getConnection();
     var statement = connection.prepareStatement(sql)) {

    statement.setString(1, entity.getName());
    statement.setBigDecimal(2, entity.getSalary());
    statement.setObject(3, entity.getBirthday());

    try (var resultSet = statement.executeQuery()) {
        if (resultSet.next()) {
            entity.setId(resultSet.getLong("id"));
        }
    }
}

Update

Atualiza name, salary e birthday pelo id. O trigger AFTER UPDATE grava os valores novos e antigos na tabela de audit. 

var sql = "UPDATE employees SET name = ?, salary = ?, birthday = ? WHERE id = ?";

statement.setString(1, entity.getName());
statement.setBigDecimal(2, entity.getSalary());
statement.setObject(3, entity.getBirthday());
statement.setLong(4, entity.getId());
statement.executeUpdate();

Delete

var sql = "DELETE FROM employees WHERE id = ?";
statement.setLong(1, id);
statement.executeUpdate();

Por enquanto não há trigger de delete — o histórico de inserts e updates permanece em employees_audit mesmo após remover o registro da tabela principal.

FindAll

SELECT * FROM employees com loop no ResultSet, montando uma List<EmployeeEntity>. O driver PostgreSQL converte TIMESTAMPTZ direto para OffsetDateTime com resultSet.getObject("birthday", OffsetDateTime.class).

FindById

Mesmo mapeamento do findAll, mas com WHERE id = ?. Retorna null se o id não existir. 

var sql = "SELECT * FROM employees WHERE id = ?";
statement.setLong(1, id);

try (var resultSet = statement.executeQuery()) {
    if (resultSet.next()) {
        var entity = new EmployeeEntity();
        entity.setId(resultSet.getLong("id"));
        entity.setName(resultSet.getString("name"));
        entity.setSalary(resultSet.getBigDecimal("salary"));
        entity.setBirthday(resultSet.getObject("birthday", OffsetDateTime.class));
        return entity;
    }
}
return null;

6. Triggers de auditoria — employees_audit

A lógica de auditoria fica no banco, não no Java: sempre que um registro é inserido ou alterado em employees, um trigger PL/pgSQL grava uma linha em employees_audit. Assim o histórico sobrevive mesmo se a aplicação mudar.

Tabela de audit (V2)

-- V2__create_employee_audit_table.sql
CREATE TABLE employees_audit (
    id BIGSERIAL PRIMARY KEY,
    employee_id BIGINT NOT NULL REFERENCES employees(id),
    name VARCHAR(150),
    old_name VARCHAR(150),
    salary DECIMAL(10, 2),
    old_salary DECIMAL(10, 2),
    birthday TIMESTAMPTZ,
    old_birthday TIMESTAMPTZ,
    operation CHAR(1),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
  • operation: 'I' para insert, 'U' para update
  • old_*: preenchidos só no update, com os valores antes da alteração
  • created_at: timestamp automático de quando o evento foi registrado

Trigger AFTER INSERT (V3)

Função fn_employee_audit_insert() + trigger tgr_employee_audit_insert. Dispara após cada linha inserida em employees. 

CREATE OR REPLACE FUNCTION fn_employee_audit_insert()
RETURNS TRIGGER AS $$
BEGIN
    INSERT INTO employees_audit (employee_id, name, salary, birthday, operation)
    VALUES (NEW.id, NEW.name, NEW.salary, NEW.birthday, 'I');
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER tgr_employee_audit_insert
    AFTER INSERT ON employees
    FOR EACH ROW
    EXECUTE FUNCTION fn_employee_audit_insert();

Trigger AFTER UPDATE (V4)

Função fn_employee_audit_update() + trigger tgr_employee_audit_update. Compara NEW e OLD e persiste ambos os lados na audit. 

CREATE OR REPLACE FUNCTION fn_employee_audit_update()
RETURNS TRIGGER AS $$
BEGIN
    INSERT INTO employees_audit (
        employee_id, name, old_name, salary, old_salary,
        birthday, old_birthday, operation)
    VALUES (
        NEW.id, NEW.name, OLD.name, NEW.salary, OLD.salary,
        NEW.birthday, OLD.birthday, 'U');
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER tgr_employee_audit_update
    AFTER UPDATE ON employees
    FOR EACH ROW
    EXECUTE FUNCTION fn_employee_audit_update();

Conferir a auditoria

-- Após insert ou update via DAO ou SQL direto:
SELECT id, employee_id, name, old_name, salary, old_salary, operation, created_at
FROM employees_audit
ORDER BY created_at;

7. Main — orquestrando tudo

A Main roda migrations, instancia o DAO e executa as operações. Trechos ficam comentados para ir testando insert, listagem e delete separadamente. 

public static void main(String[] args) {
    // 1. Flyway cria/atualiza schema, audit e triggers
    var flyway = Flyway.configure()
            .dataSource("jdbc:postgresql://localhost:5433/jdbc-sample",
                        "postgres", "050495")
            .load();
    flyway.migrate();

    // 2. CRUD via DAO (trechos comentados para testar insert, update, delete)
    var employeeDAO = new EmployeeDAO();
    employeeDAO.findAll().forEach(System.out::println);
    // employeeDAO.findById(1);
    // employeeDAO.delete(1);
}

Resumo para recrutador

Configurei um ambiente completo de persistência sem abstrações de framework: container Docker com Postgres, versionamento de schema com Flyway (tabela, audit e triggers), conexão JDBC centralizada, CRUD completo com PreparedStatement e mapeamento manual, e auditoria automática no banco via triggers PL/pgSQL. A intenção é demonstrar entendimento da camada de dados — SQL, migrations e regras no Postgres — antes de depender só de Spring Data JPA.
