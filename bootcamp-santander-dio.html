Exercícios de POO em Java

Repositório de estudos com exercícios práticos rodando no terminal. Cada desafio modela regras de negócio com orientação a objetos, separação de responsabilidades e testes automatizados onde faz sentido. Projeto em evolução — exercícios 1 e 3 concluídos; exercício 2 (carro) em andamento.

Ver código no GitHub Decisões técnicas
Java POO Gradle JUnit 5 HashMap Terminal

Como configurei o projeto

Usei Gradle como build tool, com um Main.java central que exibe um menu e direciona para cada exercício. A camada ui fica responsável apenas pela interação com o usuário (Scanner e menus); a lógica de negócio fica nas classes de domínio e serviço. 

poo-java/
├── src/main/java/org/example/
│   ├── Main.java                 # menu principal
│   ├── exercicio1/
│   │   ├── domain/ContaBancaria  # regras da conta
│   │   └── ui/Sistema            # menu do exercício
│   └── exercicio3/
│       ├── dominio/              # Pet, PetMachine
│       ├── application/          # PetShopService
│       ├── infrastructure/       # PetRepository + em memória
│       └── ui/Sistema
└── src/test/java/                # testes JUnit

Como rodar: entre na pasta poo-java e execute ./gradlew run (ou gradlew.bat run no Windows).

Exercício 1 — Conta bancária

Modelei a conta como uma única classe de domínio (ContaBancaria) com campos privados e métodos que encapsulam as regras: depósito, saque, pagamento de boleto e cálculo de cheque especial.

  • Encapsulamento: saldo e cheque especial não são alterados diretamente de fora da classe.
  • Regras no domínio: taxa de 20% ao usar cheque especial e limites de saque ficam dentro de sacar(), não no menu.
  • Exceções: valores inválidos lançam IllegalStateException; a UI apenas captura e exibe a mensagem.

Isso separa o que pode acontecer (domínio) de como o usuário interage (terminal), padrão que se repete em APIs REST depois.

Exercício 3 — Petshop (mais completo)

Este exercício foi estruturado em camadas, mais próximo de um backend real:

  • dominio: Pet e PetMachine — estado e regras da máquina de banho.
  • application: PetShopService — orquestra operações sem conhecer detalhes de entrada/saída.
  • infrastructure: persistência em memória dos pets cadastrados.
  • ui: menu interativo que chama o service.

Decisões técnicas

Por que criei a interface PetRepository?

O service precisa buscar e salvar pets, mas não deve saber como os dados são armazenados. A interface define o contrato (save, buscarPorNome, listarTodos); hoje uso implementação em memória, amanhã poderia ser banco de dados sem alterar o service. Isso é polimorfismo e inversão de dependência — base para Spring Data depois.

Por que HashMap para simular memória?

PetRepositoryEmMemoria guarda os pets em um Map<String, Pet>: busca por nome em O(1) e não exige banco neste estágio de estudo. O service continua dependendo só da interface, não do HashMap — a lista/map fica isolada na infraestrutura.

Por que constantes em PetMachine (e enum no exercício 2)?

Capacidades e consumos (água, shampoo, banho) são valores fixos do enunciado. Usei static final para centralizar e evitar números mágicos espalhados no código. No exercício 2 (carro), em desenvolvimento, pretendo usar enum para as marchas — estados limitados e bem definidos, onde enum deixa o código mais seguro que int solto.

Herança e polimorfismo — onde entram?

No exercício 3, o polimorfismo aparece via interface (PetRepository). No exercício 2, a ideia é modelar um Veículo base e Carro com herança, reutilizando comportamentos comuns e especializando regras de marcha e velocidade.

Como criei o teste unitário?

Com JUnit 5 em PetMachineTest: no @BeforeEach preparo a máquina e um pet; no teste deveDarBanhoComSucesso valido que o pet fica limpo e que água/shampoo foram consumidos conforme as constantes. Testo a regra de negócio direto no domínio, sem passar pelo menu do terminal — assim o teste é rápido e confiável. 

@Test
void deveDarBanhoComSucesso() {
    petMachine.setPet(pet);
    petMachine.takeAShower();
    assertTrue(pet.isClean());
    assertEquals(20, petMachine.getWater());
    assertEquals(8, petMachine.getShampoo());
}
